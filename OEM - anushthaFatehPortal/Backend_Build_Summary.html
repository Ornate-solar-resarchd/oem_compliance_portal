1. Executive Summary

The UnityESS Technical Compliance Portal is an enterprise-grade internal platform that manages the technical approval lifecycle of Battery Energy Storage System (BESS) component specifications across all active Ornate Agencies projects. This report documents the delivery of the complete backend implementation: 59 Python files spanning ORM models, Pydantic schemas, service classes, API endpoints, Celery workers, and Alembic migrations, built on top of the pre-existing PostgreSQL schema (17 tables, 10 enums, 12 triggers, 2 stored procedures) and fully designed React frontend (11 pages).

The backend delivers a production-ready API layer that powers compliance sheet creation, a 7-stage multi-role approval workflow, AI-assisted datasheet parameter extraction, and automated document generation across PDF, Excel, and PPTX formats. The system currently tracks 6 active BESS projects ranging from 5 MWh to 500 MWh in scale.

5. Key Business Logic Highlights

Compliance Evaluation Engine

The ParameterService evaluates OEM-submitted values against acceptance criteria defined per standard parameter. The engine supports five data types:

Data TypeEvaluation LogicExample
NumericValue within min/max boundsCycle life >= 6,000 cycles
RangeValue range overlaps acceptance rangeOperating temp: -10 to 55 C
BooleanExact match requiredUL 9540A certified: true
EnumValue in allowed setChemistry: LFP, NMC
TextPresence check or pattern matchCertification number provided

Each parameter evaluation yields pass, fail, waived, or pending. The overall sheet compliance score is computed as the ratio of passing parameters to total evaluated parameters.

System
  • Role-based transition enforcement at both application (FastAPI middleware) and database (stored procedure) levels
  • Rejection at stages 2, 3, or 4 returns the sheet to Draft with incremented revision number
  • All workflow steps are append-only (PostgreSQL trigger prevents UPDATE/DELETE)
  • Stage transitions executed via advance_workflow() stored procedure as a single atomic operation
  • SHA-256 integrity hash computed and stored at lock

AI-Powered Extraction Pipeline

  1. Engineer uploads OEM datasheet PDF via /api/v1/ai/extract
  2. File stored in MinIO; Celery task queued
  3. Claude API (claude-sonnet-4-6) extracts structured parameters based on component type schema
  4. Each extracted value receives a confidence score (0.00–1.00)
  5. Scoring thresholds: ≥0.90 auto-verified, 0.80–0.89 flagged for review, <0.80 requires manual entry
  6. Draft compliance sheet created with extracted values; notification sent to engineer

Row-Level Security

PostgreSQL RLS policies enforce data isolation at the database level. Customer users can only access sheets belonging to their assigned project, and only when the sheet has reached stage ≥ customer_submission (stage 5). Session-level variables app.user_role and app.user_id are set by the BaseService on every request.
