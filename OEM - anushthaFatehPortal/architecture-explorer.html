Architecture Pain Points
Known issues that need addressing as the system grows

All Data In Memory

Every OEM, component, deal, and RFQ lives in a single Python file (seed.py, 610 lines). All data is lost when the server restarts. No database.

Critical

No Tests

Zero test files across backend or frontend. No way to verify changes don't break existing features. High regression risk.

Critical

Monolithic Page Files

Frontend pages are 400-500 lines each. Config, forms, modals, charts, and filtering all live in a single component per page.

Warning

Direct Global State Mutation

Every API endpoint directly imports and mutates global Python lists. No service layer, no transactions, race conditions possible.

Warning

No Type Contracts

API returns untyped Python dicts. Frontend defines the same TypeScript interfaces in multiple files. Type mismatches caught only at runtime.

Warning

Duplicated Config

OEM colors, stage configs, and icon mappings appear in 5+ different frontend files. Changes require updating multiple locations.

Where We Go Next
Architecture Options
Four evaluated paths for evolving the portal from POC to production
Option 1

Layered Services

Add a service layer between endpoints and data, plus a repository pattern for swappable storage. Frontend gets shared types and custom hooks.

Strengths

  • Best database migration path
  • Incremental — one domain at a time
  • Services testable in isolation
  • Right-sized for small team

Tradeoffs

  • More files to navigate
  • Repository pattern learning curve
  • Discipline needed to not bypass layers
Option 2

Thin Backend

Strip backend to pure CRUD + AI extraction. Move all business logic to the frontend with React Query for caching.

Strengths

  • Backend becomes trivially simple
  • React Query eliminates boilerplate
  • Fewer endpoints to maintain

Tradeoffs

  • Frontend bundle grows significantly
  • Complex calculations in browser
  • Can't share logic with mobile app
Option 3

Domain Modules

Reorganize into vertical domain modules (oem/, rfq/, pipeline/) each containing router, service, types, and seed data.

Strengths

  • Strong domain isolation
  • Each domain independently testable
  • Great for larger teams

Tradeoffs

  • Massive file reorganization
  • Over-engineered for 1-2 devs
  • Cross-domain queries become harder
Option 4

Do Less

Fix only the worst pain points: split seed.py, add response schemas, extract frontend types, add React Query.

Strengths

  • Minimal disruption
  • Zero learning curve
  • Pure wins, nothing gets harder

Tradeoffs

  • Structural problems remain
  • No database migration path
  • Same friction adding features

Recommended Roadmap
Option 1 (Layered Services), executed incrementally starting with quick wins

Quick Wins & Hygiene Week 1

Low-risk improvements that pay off immediately

  • Split seed.py into 4 domain files (OEM, component, pipeline, RFQ)
  • Add Pydantic response models to all API endpoints
  • Extract frontend TypeScript types to shared types/ directory
  • Add React Query to 1-2 pages (Pipeline, OEMs)

Service Layer (Core Domains) Week 2-3

Add services for the most active domains

  • Create PipelineService and RFQService with business logic
  • Extract workflow state machine into WorkflowService
  • Endpoints become thin controllers
  • Add structured logging (replace print statements)

Repository Pattern Week 4

Abstract data access for future database migration

  • Define repository interfaces per domain
  • Implement in-memory repositories (wrap existing seed data)
  • Swap all services to use repositories instead of direct imports
  • Verify demo mode still works with seed data

Frontend Polish & Database Prep Week 5+

Component extraction and production readiness

  • Extract modal/dialog components from page files
  • Extract chart components for reuse
  • Consolidate config objects (OEM_INFO, STAGE_CONFIG)
  • Implement PostgreSQL repositories (swap in for in-memory)
